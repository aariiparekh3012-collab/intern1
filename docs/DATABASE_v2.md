# Database Design v2 — Platform Schema (Multi-Module)

Companion to `ER_DIAGRAM_v2.mermaid`. This documents the refactored schema that extends the
onboarding module to support the full discretionary PMS platform. **Onboarding tables are
unchanged.** Everything else is additive and organised into PostgreSQL **schemas** (namespaces)
for clean grants and blast-radius isolation.

## Core principle

Onboarding is a *transient funnel*. On the `ACTIVE` event it **provisions** a durable
`client.clients` record (the golden record). All downstream modules attach to `clients` and
`portfolio_accounts` — never to `onboarding_applications`. A client may hold multiple portfolio
accounts (one per strategy), so the client↔account split is fundamental.

Ledgers (`trades`, `capital_flows`, `cash_ledger`, `corporate_actions`) are the **immutable
source of truth**; `holdings` and `valuation_snapshots` are **derived/materialised** for fast
reporting.

Conventions: UUID PKs · `timestamptz` (UTC) · money as integer **paise** (`BIGINT`) ·
quantities/percentages as `NUMERIC` · PII encrypted at rest, identifiers hashed for lookup.

## Schemas & tables

### `onboarding` (unchanged)
- `onboarding_applications`, `onboarding_documents`

### `client` — Client Master
- `clients` — golden record; FK back to source `onboarding_application_id`; `client_code` UK
- `client_bank_accounts` — many per client; one `is_primary`
- `client_demat_accounts` — NSDL/CDSL BO accounts
- `nominees` — up to 3, `share_percent`, `rank` (SEBI nomination)
- `client_risk_profiles` — versioned over time; onboarding seeds v1, re-profiling appends

### `reference` — Master / static data
- `securities_master` — ISIN (UK), symbol, exchange, instrument_type, sector
- `strategies` — discretionary model portfolios; FK `benchmark_id`
- `strategy_constituents` — target model weights (rebalancing/drift)
- `benchmarks`, `benchmark_values` — index definitions + daily levels
- `corporate_actions` — dividend / split / bonus
- `brokers` — empanelled brokers (SEBI reg no)

### `portfolio` — Portfolio Accounts + Holdings
- `portfolio_accounts` — client × strategy × demat; `account_code` UK; central entity
- `fee_schedules` — mgmt %, perf %, high-water-mark (fee module placeholder)
- `capital_flows` — corpus contributions/withdrawals (cash or securities)
- `cash_ledger` — running cash movements
- `holdings` — current position per account/security (qty, avg cost) — *derived*
- `holding_lots` — FIFO tax lots for capital-gains

### `trading` — Trades
- `orders` — discretionary order generated by PM at strategy level
- `order_allocations` — block order split pro-rata across accounts
- `trades` — executions/fills booked per account, with broker + price

### `performance` — Performance Reporting
- `valuation_snapshots` — daily account market value vs cost — *derived*
- `performance_returns` — TWRR by period (MTD/QTD/YTD/ITD) vs benchmark
- `statements` — generated client statement metadata (PDF in object storage)

### `compliance` — Compliance Reviews
- `compliance_reviews` — periodic / re-KYC / suitability, maker-checker
- `suitability_assessments` — risk profile ↔ strategy fit
- `compliance_alerts` — concentration / mandate-drift / cash-breach exceptions

### `platform` (shared, unchanged structure)
- `audit_logs` — append-only (INSERT/SELECT grants only)
- `event_outbox` — transactional outbox for cross-module events

## Module → table mapping (requested modules)

| Module | Primary tables |
|--------|----------------|
| Client Master | `client.clients`, `client_bank_accounts`, `client_demat_accounts`, `client_risk_profiles` |
| Portfolio Accounts | `portfolio.portfolio_accounts`, `fee_schedules`, `capital_flows`, `cash_ledger` |
| Holdings | `portfolio.holdings`, `holding_lots` |
| Trades | `trading.orders`, `order_allocations`, `trades`; `reference.brokers`, `securities_master` |
| Performance Reporting | `performance.valuation_snapshots`, `performance_returns`, `statements`; `reference.benchmarks`, `benchmark_values` |
| Nominees | `client.nominees` |
| Compliance Reviews | `compliance.compliance_reviews`, `suitability_assessments`, `compliance_alerts` |

## Key design decisions (for sign-off)
- **Non-unitized** accounting (direct securities + TWRR) — standard for Indian discretionary PMS.
- **Ledger-as-truth**; holdings & valuations are rebuildable projections.
- **Schema-per-bounded-context** for least-privilege grants and independent migrations.
- **PII protection** carried forward from v1: encryption at rest + hashed lookup keys + masking.
- Fee/billing modelled as a placeholder (`fee_schedules`) so the future fee module slots in.
